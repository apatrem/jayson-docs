# Document System — Architecture Design Memo

**Audience:** the developer who will build and own this system
**Status:** design proposal, not yet approved for build
**Date:** 2026-05-21
**Build approach:** greenfield — built from scratch; no code, schema, or configuration carried over from any prior prototype

---

## 1. Purpose & build approach

Build, from scratch, an internal system that lets consultants
(non-developers) produce client deliverables — flowing documents, slide decks,
and interactive HTML — that are visually consistent, anchored on the company
template, partly generated by an LLM, and editable in a WYSIWYG surface.

This memo describes the **target architecture**. It is the result of a scoping
discussion; the requirements in §3 are fixed inputs, not open questions.

### Clean slate
This is a **greenfield build**. The developer starts from an empty repository.
No code, schema, configuration, or component is imported from any earlier
experiment. Prior prototypes may be looked at for ideas, but nothing is carried
over — the system is built to this memo, not migrated from anything.

### Templating inputs — demo DOCX/PPTX
The consultancy will supply a few demo DOCX, PPTX, and (optionally) PDF brand
materials. Their role is deliberately narrow and one-time:

- They are used **once, at setup**, to derive the **brand identity**
  (colours, fonts, logo, spacing — see `brand.example.yaml`) and to inform
  **which blocks the consultancy needs** (see `blocks.catalogue.yaml`).
- They are processed by a **setup-time AI pipeline** that produces two
  artifacts for human review:
  1. A draft brand-token file populated from the observed visual identity.
  2. A catalogue diff: which of the 15 pre-built Standard blocks the
     consultancy uses + proposals for any Brand blocks that need to be
     generated (subject to the Layer 1 mitigations; see ADR-0004 for the
     Standard / Brand / Authored three-tier taxonomy).
- Both artifacts land in a **pending review state** and require human
  approval before the app will load them.
- The demo files **do not influence the architecture.** The architecture in
  this memo is fixed and stands independent of whichever demo files are
  supplied. Only the brand-token file and the block catalogue (data, not
  architecture) vary per consultancy.
- **Do not build runtime DOCX/PPTX parsing or import into the system.** The
  setup pipeline is the only place Office files are read, and it runs
  out-of-band of the editor/renderer (consistent with R3 and R15). The
  runtime system never reads or writes Office formats.

## 2. The one principle that governs everything

**The source of truth is a schema-defined document model
("DocModel"). The editor, the collaboration layer, and YAML are all
*projections* of it — none of them is canonical.**

A common mistake (made by at least one AI design suggestion reviewed during
scoping) is to make the editor's internal state (TipTap/ProseMirror JSON, or a
Yjs CRDT document) the canonical artifact. Do not do this. It causes:

- **Editor lock-in** — every document becomes trapped in an
  editor-specific format.
- **Lossy round-trips** — YAML ⇄ editor-JSON ⇄ YAML is not a clean bijection;
  documents degrade on each conversion.
- **Broken comment anchoring** — AI comments need stable anchors that survive
  serialization; if comments live only in editor state they die on export.
- **The loss of the insurance policy** — a canonical plain file must render to
  PDF/HTML *even if the editor never runs*.

The DocModel is canonical. TipTap edits it. A renderer consumes it. The LLM
patches it. YAML is a readable projection of it. Yjs (later) syncs it.

## 3. Requirements & constraints (fixed)

| # | Requirement | Decision |
|---|-------------|----------|
| R1 | Scope | Flowing documents **and** slide decks **and** interactive HTML |
| R2 | Docs vs decks | **Separate artifacts** sharing a common component/brand library |
| R3 | Client handoff | **Read-only PDF / HTML.** No editable Office round-trip required |
| R4 | Consultant editing | **WYSIWYG only.** Consultants never see YAML/JSON/code |
| R5 | Editor capability | Edit text & numbers; reorder/add/remove blocks from a library; limited element placement on a **fixed grid** |
| R6 | Template consistency | Enforced **structurally** — closed block library + schema, not by review |
| R7 | LLM generation | Starts from a **structured outline**; fills blocks |
| R8 | Chart data | **Typed in the editor or pasted from Excel** — static snapshots, no live data link |
| R9 | Charts | ECharts-class quality is adequate; **not** a think-cell clone |
| R10 | Vendors | **Open-source tools only** for the build/runtime stack; LLM via a vetted vendor (see §9) |
| R11 | Volume | 40+ client documents/month across ~30 consultants |
| R12 | Ownership | A hired developer maintains the system |
| R13 | Feature | Highlight text + leave a comment to the AI for updates, in the WYSIWYG editor (see §7) |
| R14 | Build approach | **Greenfield.** Built from scratch; no code/schema/config reused from any prototype |
| R15 | Office files | Demo DOCX/PPTX are a one-time **templating input** only; the system never parses or generates Office formats at runtime |

**Open decisions still to resolve before build (see §11).**

## 4. Target architecture

```
┌─────────────────────────────────────────────────────────────┐
│ LAYER 6 — OUTPUT                                              │
│   HTML (primary) · PDF (secondary) · interactive HTML (later) │
│   NO PPTX export                                              │
├─────────────────────────────────────────────────────────────┤
│ LAYER 5 — RENDERING                                           │
│   DocModel → renderers (React components, one per block       │
│   type) → HTML. PDF via headless Chromium (Playwright).       │
│   Deck = separate renderer over the DocModel, with a          │
│   fixed library of slide layouts. Charts via ECharts.         │
├─────────────────────────────────────────────────────────────┤
│ LAYER 4 — COLLABORATION & HISTORY        (DEFER — design for) │
│   v1: history from DocModel being a plain versioned file.     │
│   Later: Yjs syncing the DocModel via a self-hosted server.   │
├─────────────────────────────────────────────────────────────┤
│ LAYER 3 — EDITING SURFACE (WYSIWYG)                           │
│   TipTap open-source core (on ProseMirror).                   │
│   Custom nodes per block type. Maps DocModel ⇄ editor doc.    │
│   Block library + grid anchoring. Hosts comments.             │
├─────────────────────────────────────────────────────────────┤
│ LAYER 2 — LLM INTERFACE                                       │
│   Generates/edits the DocModel via JSON structured output.    │
│   Input = structured outline. Edits = scoped block patches.   │
├─────────────────────────────────────────────────────────────┤
│ LAYER 1 — SCHEMA & VALIDATION                                 │
│   One Zod schema = the company template as code.              │
│   Closed block library, allowed layouts, brand tokens.        │
├─────────────────────────────────────────────────────────────┤
│ LAYER 0 — CANONICAL DOCUMENT MODEL (DocModel)                 │
│   One schema-defined JSON document. THE source of truth.      │
│   Renders without the editor. YAML is a projection of it.     │
└─────────────────────────────────────────────────────────────┘
```

### Layer 0 — Canonical DocModel
A single JSON document conforming to the schema. Contains: document metadata,
a brand reference, an ordered list of sections, each with an ordered list of
typed blocks. **Every block carries a stable `id`** (required for comment
anchoring and for scoped LLM patches). Rich text inside prose blocks is stored
as a ProseMirror-JSON fragment so text editing is lossless. YAML is generated
from / parsed to the DocModel as a human- and LLM-readable projection — first
class, but not canonical.

### Layer 1 — Schema & validation
**Define a Zod schema from scratch.** It is used in three places: validating
LLM output before it is accepted, constraining what the editor can build, and
gating rendering. **This schema is the implementation of "anchored on the
company template" (R6).**

**The block library is closed *at any given moment*, with a three-tier
composition (see ADR-0004):**

- **Standard blocks (Tier 1 — 15 blocks).** Ship with the app.
  Built once by the developer; spec lives in `blocks.catalogue.yaml`
  (prose, heading, bullet-list, numbered-list, chart, table, callout,
  kpi-cards, timeline, roadmap, risk-matrix, team, image, diagram, divider).
  Brand-token mapping makes these feel native to each consultancy without
  code changes.
- **Brand blocks (Tier 2).** Added at setup time by an AI that scans the
  consultancy's demo files and generates new blocks *only when an observed
  pattern cannot be expressed by Standard blocks with brand-token mapping
  alone*. Sized at approximately 15 per consultancy by observation.
- **Authored blocks (Tier 3).** Generated on demand by individual
  consultants and shareable peer-to-peer. Auto-install after passing the
  lint + runtime watchdog gate; no human review required. See ADR-0004 for
  the full lifecycle model.

**Mitigations on Brand blocks (mandatory; Authored blocks follow ADR-0004):**
1. **Constrained template** — AI fills a pre-defined node-view scaffold;
   no free-form React.
2. **Whitelisted imports only** — `react`, `@tiptap/*`, `echarts/*`,
   `./brand-tokens`, `./block-primitives`. Lint rule fails the build on
   imports outside this set.
3. **Forbidden patterns** — no `dangerouslySetInnerHTML`, `eval`,
   `Function`, `fetch`, `XMLHttpRequest`, or direct CSS injection.
4. **Human review gate** — generated files land in `/generated-blocks/pending/`;
   the app refuses to load them until a human reviewer moves them to
   `/generated-blocks/active/`.
5. **CSP-sandboxed rendering** — generated blocks render inside a
   Content-Security-Policy-constrained iframe in the editor; pre-rendered
   to static HTML at PDF export time.
6. **Regen pipeline** — when the host app updates the scaffold, a
   `regenerate` script re-runs generation and routes any drift to
   `/pending/` for re-review.

**The library is closed at any given moment**; it evolves as consultants
generate and share Authored blocks. A consultant cannot produce an
off-template document because off-template blocks do not exist in the
active library.

There must be exactly one machine-readable brand source (colours, fonts,
spacing), derived from the demo DOCX/PPTX at setup (§1) and consumed by every
renderer. If the documented brand and actual template files disagree,
resolve the conflict into that single authoritative source.

### Layer 2 — LLM interface
The LLM generates and edits the DocModel. Generation starts from a structured
outline (R7) and fills blocks. **All LLM edits are structured patches scoped to
specific blocks — never full-document rewrites** — so human edits are never
clobbered. Prefer JSON structured output (the schema can constrain decoding);
YAML is fine as a readable equivalent.

### Layer 3 — Editing surface
See §6 and §7.

### Layer 4 — Collaboration & history
**Deferred for v1.** History is free in v1 because the DocModel is a plain
versioned file (Git or equivalent). When real-time co-editing is genuinely
needed, add Yjs — syncing the DocModel, not becoming a separate source of
truth — with a self-hosted server (Hocuspocus, MIT). Designing Layer 0 as
canonical now keeps this path open without committing to it.

### Layer 5 — Rendering
Render **from the DocModel**, not from editor JSON — this keeps "renders
without the editor" true. **Build the block renderer components from scratch**:
one React component per block type (chart, table, callout, timeline, KPI cards,
etc.), each consuming one block of the DocModel.

**PDF: render the HTML with headless Chromium via Playwright — one engine.**
HTML is the primary output, so PDF is simply "print the HTML": the *same*
Chromium engine renders the HTML deliverable and prints the PDF, so the two
cannot diverge. Page breaks use native CSS (`break-inside: avoid` to keep a
chart, table or heading-block together); running headers, footers and page
numbers use Playwright's `headerTemplate` / `footerTemplate` options on
`page.pdf()` — no third-party pagination library required.

**WeasyPrint is a fallback, not the default.** It is a separate, non-browser
rendering engine with its own layout implementation: it would render the PDF
differently from the browser that renders the HTML (a milder version of the
divergence problem), and it runs **no JavaScript**, so ECharts charts and
Mermaid diagrams would have to be pre-rendered to static SVG before it. Adopt
it only if documents later prove to need book-grade pagination — footnotes,
complex margin-box content — that Chromium genuinely cannot produce. **Typst was
rejected** for the stronger version of the same reason: a separate typesetting
engine consuming neither HTML nor the DocModel — a full second renderer.

The **deck** is a separate renderer over the same DocModel, using a *fixed
library of slide layouts* (title, two-column, chart-full, section divider).
Keep renderers thin and swappable — they are the part most exposed to
obsolescence if better open tools appear.

### Layer 6 — Output
HTML first, PDF second. Interactive HTML with embedded live models is a v2
feature (§10). **No PPTX export**, not even as an escape hatch — it reintroduces
the entire OOXML problem and is a consistency leak; client output is read-only
PDF/HTML by requirement R3.

## 6. The editor: TipTap on ProseMirror

Use **TipTap's open-source core**, which is MIT-licensed and built on
ProseMirror. Do not use raw ProseMirror alone (it only adds boilerplate for no
licensing benefit) and do not use Tiptap Pro/Cloud (paid, fails R10).

The editor maps the DocModel to a TipTap document and back:

- **Structural blocks** (chart, table, KPI cards, callout, risk matrix,
  roadmap, etc. — the definitive list comes from the demo-file templating step,
  §1) → custom TipTap **atom/leaf nodes** whose data lives in a schema-validated
  `attrs` payload, edited through a React node view (a small form / data grid),
  not as free text.
- **Prose blocks** → native ProseMirror rich text.
- The **DocModel ⇄ editor-document mapping must be explicit and tested for
  losslessness in both directions.** This is real engineering work — budget
  for it; it is not a trivial conversion.
- Editing is **block-library + grid-anchored** (R5): consultants add blocks
  from a closed palette and place a limited set of elements on a fixed grid.
  No free canvas.

## 7. Feature: highlight + comment-to-AI

A consultant highlights text in the WYSIWYG editor, attaches an instruction for
the AI ("quantify this", "shorten to one line", "make this title action-
oriented"), and the AI proposes an edit the consultant can accept or reject.

Because TipTap's Comments extension is paid (fails R10), the comment system is
**built in-house** on ProseMirror primitives: a custom inline **mark** with an
ID for the highlighted range (marks move with content as text changes), plus a
comment record stored **on the DocModel**, not in editor state.

### Comment record (stored in a top-level `comments[]` array on the DocModel)
```
Comment {
  id:          StableId
  blockId:     StableId                    // stable anchor — survives export
  range:       { from: int, to: int } | null // char offsets within the block
  quotedText:  string                      // snapshot of highlighted text
  status:      "open" | "applied" | "rejected"
  thread:      ThreadEntry[]               // instruction → ai-proposal → follow-up ...
  createdAt:   timestamp
  updatedAt:   timestamp
}

ThreadEntry =
  | { kind: "instruction", author, authorEmail, authorRole, text, createdAt }
  | { kind: "ai-proposal", model, patch: BlockPatch, inputTokens, outputTokens, createdAt }
  | { kind: "follow-up", author, authorEmail, text, createdAt }
}
```

### Workflow
1. Consultant highlights text and writes an instruction.
2. On trigger, the system sends the AI: the targeted block(s), the highlighted
   span, and the instruction — with the rest of the document as **read-only
   context**.
3. The AI returns a **structured patch scoped to that block** as an
   `ai-proposal` thread entry.
4. The patch is displayed as a **proposed change (track-changes style)** — it
   is **never auto-applied**.
5. The consultant accepts or rejects. On accept, the patch is applied as one
   structured operation to one block; the comment resolves.

### Four rules that keep this safe
- **Scope strictly** to the comment's block/range. A comment must never trigger
  a whole-document regeneration.
- **Proposals, never auto-apply.** This is an internal review surface for the
  consultant's own AI edits — it mirrors Word track-changes, which consultants
  already understand. It does not conflict with read-only client output (R3):
  nothing here is exposed to the client.
- **Comments anchor to stable `blockId`s** so they survive YAML serialization
  and the editor↔DocModel mapping. (This is why Layer 0 must be canonical.)
- **Support a batch mode.** Consultants mark up a whole draft with several
  comments, then "process all" and review every proposed change at once — the
  real consulting workflow (marking up a junior's draft). This is the feature's
  main value.

This feature needs **no real-time collaboration** — comments are async
annotations — so it ships fully in v1 without Yjs.

## 8. Component build list

Everything below is built from scratch. There is no pre-existing code to reuse.

| Component | Phase |
|-----------|-------|
| Zod schema + DocModel definition (incl. stable block IDs) | v1 |
| Brand-token file + block/layout catalogue (derived from demo DOCX/PPTX, §1) | v1 setup |
| Block renderer components — one React component per block type | v1 |
| HTML renderer (DocModel → HTML) | v1 |
| PDF export (Playwright; headers/footers via `page.pdf()` templates) | v1 |
| LLM interface (outline-driven generation; scoped block patches) | v1 |
| Chart blocks via ECharts; diagram blocks via Mermaid | v1 |
| TipTap editor with custom nodes per block type | v1 |
| DocModel ⇄ editor-document mapping (+ losslessness tests) | v1 |
| In-house comment system + comment-to-AI workflow | v1 |
| Deck renderer + fixed slide-layout library | v1.1 |
| Collaboration (Yjs + Hocuspocus); interactive HTML / live models | v2 |

The first build is the **flowing-document path, end to end**: schema → LLM
generation → editor → comment-to-AI → HTML/PDF. The deck path then reuses the
DocModel, schema, editor and comment system; only a deck renderer and a
slide-layout library are added on top.

## 9. The LLM (the one non-open-source component)

The build/runtime stack is fully open source. The LLM is the exception.

- **Recommended:** a hosted frontier model via API from a vetted vendor
  (Anthropic/Claude has already been accepted as a vetted vendor; client
  content may transit it). Quality for structured, outline-driven consulting
  documents is materially better than open-weights alternatives today.
- **Open-source-purist option:** self-host an open-weights model (Llama,
  Mistral, Qwen). This keeps 100% of the stack open and on-premise, but expect
  a **meaningful quality penalty** for structured document generation and
  instruction-following. Not recommended unless data policy hardens.
- The architecture is **model-agnostic**: the LLM only reads/writes the
  DocModel via the Layer 2 interface, so the model can be swapped without
  touching the editor or renderers.

## 10. Non-goals (explicitly out of scope)

Building any of these is how this kind of project fails. Do not build them:

- **A think-cell clone.** Chart quality was not flagged as a pain. ECharts is
  adequate. Add 3–4 consulting chart types (waterfall, Mekko, stacked bar with
  CAGR arrow) incrementally — nothing more.
- **A free-canvas deck editor.** Slides are a *fixed library of layouts* a
  consultant fills. Free positioning destroys consistency and rebuilds
  PowerPoint badly.
- **DOCX/PPTX import or parsing.** Demo Office files are a design-time
  reference only (R15). The system never touches Office formats at runtime.
- **PPTX export.** See Layer 6.
- **Real-time collaboration in v1.** Deferred (Layer 4).
- **A general "live models" platform.** Interactive HTML with embedded live
  models is a v2 feature; when first needed, hand-build it for that one
  deliverable rather than generalising prematurely.

## 11. Open decisions (resolve before build starts)

1. **Budget.** "A hired developer" + "budget undecided" is a contradiction. A
   developer building DocModel + editor + comment system + deck renderer is a
   6–12 month commitment. The business case is makeable (40+ docs/month; even
   2 hours saved per doc ≈ a half-FTE recovered) but must be written and
   approved before a hire is real.
2. **Doc-heavy vs deck-heavy.** Is the 40+/month volume mostly flowing
   documents or mostly decks? If deck-heavy, the roadmap below must be
   re-ordered so the deck renderer comes first.
3. **Half-day technical scan.** Before committing, the developer scans editor
   libraries and the deck-rendering approach (§12 step 1). Document/PDF
   rendering is already decided (Playwright, §4–§5).

---

## 12. Roadmap

1. **Half-day technical scan** — editor libraries and the deck-rendering
   approach. Document/PDF rendering is already decided: Playwright (§4–§5).
2. **Business case + budget sign-off.** Without it, stop here.
3. **v1 — flowing documents only** (proposals, reports): Zod schema + DocModel,
   LLM outline-driven generation, block renderers, HTML/PDF output, TipTap
   editor with custom nodes, DocModel⇄editor mapping, in-house comment-to-AI
   feature. Ship to a few consultants and prove the full path before adding
   decks.
4. **v1.1 — Deck renderer** + fixed slide-layout library.
5. **v2 — Collaboration** (Yjs + Hocuspocus) and **interactive HTML / live
   models**, only once a concrete need appears.

Running steps 3–5 in parallel is the main risk to shipping. Do them in order.

---

## 13. Key risks

- **Single-maintainer rot** — a bespoke internal tool whose only developer
  leaves. Mitigations: keep the stack boring and standard (everything in §5 is
  mainstream); keep the DocModel an open, portable, schema-validated file that
  renders without the editor. The DocModel-as-canonical principle (§2) *is* the
  insurance policy — protect it absolutely.
- **Scope creep into the §10 non-goals** — the most likely failure mode.
- **DocModel⇄editor mapping underestimated** — it is real work; budget for it
  and test losslessness explicitly.
